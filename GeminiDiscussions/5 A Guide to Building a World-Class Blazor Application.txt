A Guide to Building a World-Class Blazor Application
Our conversation detailed the complete architectural journey of building a modern, scalable, and high-performance Blazor application. It began with a simple question about creating separate desktop and mobile page versions and evolved into a deep exploration of world-class software engineering principles, including project structure, dependency management, domain-driven design, and security. This narrative documents that entire trajectory, including the reasoning behind every decision, refinement, and change of direction.
The initial goal was to display different user interfaces for desktop and mobile users. The first solution proposed was to use a device detection service. This involved creating a small JavaScript file in the project's wwwroot folder with a function called getWindowWidth to return the browser's inner width. This function would be called from C# code using Blazor's JS Interop capabilities. A C# service, which we called DeviceDetectionService, would be created to wrap this JS Interop call. This service could then be injected into any Razor component. A "router" page component would inject this service, call it to determine if the device was mobile based on a pixel breakpoint, and then use a simple if-else block to render either a MyPageDesktop.razor component or a MyPageMobile.razor component. This approach was presented as a modern alternative to older methods like using separate "m.domain.com" websites, as it keeps a single URL for better SEO and maintenance while allowing for completely different UI structures, unlike CSS media queries which are better suited for stylistic changes.
The user then asked how to apply this same pattern to the application's main layout, to have a different navigation structure for desktop and mobile. The advice was to extend the pattern by creating a "layout router" component. This new component would inherit from Blazor's LayoutComponentBase class. It would inject the same DeviceDetectionService and, in its markup, use an if-else block to render either a DesktopLayout.razor or a MobileLayout.razor component, passing the special Body parameter to whichever layout was chosen. This layout router component would then be set as the DefaultLayout in the application's root Routes.razor component, effectively wrapping every page in this device-aware logic.
This led to questions about compatibility. For the "Request Desktop Site" feature in mobile browsers, it was explained that the proposed solution using window.innerWidth would work perfectly. When a user requests the desktop site, the browser reports a wider, virtual viewport width, which the DeviceDetectionService would correctly interpret as a desktop device, serving the desktop layout as expected. For compatibility with a .NET MAUI application, it was clarified that the JS Interop approach would not work because MAUI apps run in a native process without a browser's window object. The correct pattern for MAUI would be to use its built-in C# APIs, specifically the Microsoft.Maui.Devices.DeviceInfo class, to determine the device idiom, such as Phone or Tablet.
This distinction prompted an insightful question from the user about using an interface to abstract the device detection logic. This was identified as a world-class best practice. The proposed solution was to create an IDeviceDetectionService interface in a shared project. Then, two implementations would be created: a WebDeviceDetectionService in the main web server project that used JS Interop, and a MauiDeviceDetectionService in a future MAUI project that used the native MAUI APIs. The application's UI components would only depend on the interface, making them completely platform-agnostic.
The conversation then shifted to the overall project structure. The user had a solution with a server project named TheCanalaveLibrary, a client project for WebAssembly, and a shared UI library called TheCanalaveLibrary.SharedUI. The user asked if all Razor components should be moved to the SharedUI project. The advice was an emphatic yes. All reusable UI components should be placed in the SharedUI library to allow them to be used with any of Blazor's render modes, whether server-side or on WebAssembly, and to keep the main server project lean and focused on hosting and services. Core application shell components like App.razor and Routes.razor would remain in the server project.
Next, we addressed the data layer structure. The user had their interfaces and DTOs in a TheCanalaveLibrary.Core project and their Entity Framework model classes in the main server project. The initial advice was to also move the EF Core model classes into the Core project. The reasoning was that this would maximize code reuse, allowing both the client and server to understand the shape of the data entities, thereby reducing the need to create DTOs that were just copies of the models.
This advice was met with correct and insightful pushback from the user, who raised critical concerns about the dangers of using EF Core models directly across the client-server boundary, such as issues with EF's change tracker and security vulnerabilities like over-posting. This was a pivotal moment in the conversation. The advice was immediately refined to embrace a more robust, world-class hybrid approach. The EF models should still be moved to the Core project, but a strict rule must be followed: service interfaces must only ever use DTOs. The server-side service implementations would be responsible for querying the database using the EF models and then projecting the results into DTOs before returning them to the client. This maintains a clean and secure API boundary.
The user then, quite rightly, questioned why the models needed to be in the Core project at all if the client only ever interacts with DTOs. The justification for this "pragmatic" architecture was explained. Having both the model classes and the DTOs available in the shared Core library enables powerful patterns. The primary benefit is the ability to define shared projection logic. For example, an AutoMapper mapping profile, which defines how to convert a Story model to a StorySummaryDto, could be created in the Core project. This same profile could then be used by the server to efficiently project database queries and by the client to perform in-memory mapping, ensuring the logic is defined only once.
Another proposed benefit was sharing validation attributes. The first example given to demonstrate this involved having a DTO inherit from an EF model class. The user correctly identified this as a flawed pattern, as it would make the DTO heavyweight and bloated with unwanted navigation properties. This led to a much better solution: using a shared validation interface. An interface, for example IChapterValidation, would be created in the Core project containing properties decorated with validation attributes. Both the lightweight DTO and the full EF model class would then implement this interface, creating a single, verifiable contract for validation rules without forcing inheritance.
The user remained unconvinced, pointing out that even with an interface, validation attributes would need to be duplicated on the model class for EF Core Migrations to work. This led to the final and most important justification for the shared model pattern: enabling a Rich Domain Model with Behavior. This is a concept from Domain-Driven Design where the model classes themselves contain business logic. Instead of being anemic property bags, the models have methods like story.Publish() or story.AddChapter(). The key insight was that with Blazor, this C# logic, defined once in the Core project, could be executed on the server before saving to the database to ensure data integrity, and also executed on the client in WebAssembly to provide instant UI feedback to the user. This solves the classic web development problem of having to duplicate business rules in both a server-side language like C# and a client-side language like JavaScript. This pattern is most useful for complex "write" workloads, while simple "read" workloads should still use lightweight, projected DTOs for performance.
The conversation then moved to a series of practical implementation details based on this final architecture. For the user's planned "Tree Search" graph traversal feature, it was confirmed that this was a perfect use case for the rich domain model, where the client would maintain an in-memory graph of rich model objects, lazy-loading additional data from the server via DTOs as needed. We discussed managing risks, such as the need for null checks on navigation properties in Razor components, keeping the WASM payload low by using specific DTOs for read operations, and avoiding "leaky abstractions" by ensuring no server-specific dependencies like DbContext were ever used in a shared model class. For EF Core Migrations, it was clarified that adding methods and non-mapped properties to models is safe, while changing the data schema is not. The correct command-line syntax for running migrations with this distributed project structure was also provided. We also covered how EF Core is able to set properties with private setters by writing directly to their backing fields, and how it uses parameterless constructors to create object instances.
To optimize a critical search query based on the IsPubliclyVisible property, a filtered index was recommended as a superior solution to denormalization, as it offers high performance without the risk of data synchronization issues. The EF Core configuration for creating such an index in the ApplicationDbContext was detailed.
The discussion then addressed the placement of ASP.NET Core Identity models. It was recommended to move the User and ApplicationRole classes to the Core project, accepting the framework dependency on Microsoft.Extensions.Identity.Stores in exchange for a complete and cohesive domain model. The small increase in the WebAssembly payload size was deemed an acceptable trade-off due to the .NET linker's ability to trim unused code. A specific issue with a database-specific type, NpgsqlTsVector, in a model class was resolved by using EF Core's "shadow properties" feature, which keeps the C# model clean while configuring the database-specific column entirely within the ApplicationDbContext.
The topic of caching with Redis was explored. It was established that clients must never connect directly to Redis for security reasons. The correct pattern is for the server to act as a trusted intermediary. For server-rendered components, Redis can be used directly by injecting IDistributedCache. For client-side WebAssembly components, they should call a server API endpoint, and that endpoint would be responsible for the caching logic. The idea of creating an HTTP-based implementation of IDistributedCache for the client was identified as an anti-pattern due to extreme performance overhead and security complexities. The superior alternative, the "Cached Repository" or "Decorator" pattern, was introduced, where a CachedStoryService wraps the real StoryService on the server to handle caching transparently.
This led to the final major architectural challenge: how a single Razor component could work during server-side prerendering (where it can access Redis) and on the client with WebAssembly (where it cannot). The definitive solution presented was to use Blazor's PersistentComponentState. The component would be configured with two different dependency-injected implementations of its service interface. During prerendering on the server, it would use the Redis-backed service to fetch data and then use PersistentComponentState to serialize and embed that data into the rendered HTML. When the Blazor WebAssembly runtime starts on the client, the component would initialize again, but this time it would find and deserialize the data from the persisted state, completely avoiding a redundant API call.
The final phase of our conversation involved the practical difficulties of refactoring the default scaffolded Identity UI. The user moved all Identity components to the SharedUI project but encountered compilation errors because many of those components, like Register.razor, directly inject server-only services such as UserManager and HttpContext. The initial advice was to move these server-dependent components back to a dedicated folder in the server project, as this is the standard and most pragmatic architecture. The user felt this invalidated their refactoring effort. An alternative was proposed: refactoring the Identity pages themselves to call custom-built API endpoints instead of injecting services directly. However, it was explained that abstracting a complex security framework service like UserManager over HTTP is exceptionally difficult and not recommended due to its large API surface and its deep, non-abstractable integration with the server's cookie authentication pipeline.
This brought the conversation full circle, solidifying the conclusion that the most robust, secure, and industry-standard architecture is a hybrid one. The core Identity pages that perform secure operations must remain as statically-rendered components on the server. The truly universal UI components, such as layouts, navigation, and interactive application pages, should reside in a pure, shared UI library. This was not a failure of the refactoring process, but rather the successful discovery of the application's true architectural boundaries. The final guidance was to proceed with this clean separation, which provides a powerful and maintainable foundation for all future development.
