Story Creation and Editing Pipeline Refinement
Our conversation began with the fundamental architectural challenge of creating a single user interface page in Shared UI that could serve a dual purpose: both creating a new story and editing an existing one. The initial request sought different approaches to implementing this feature, focusing on Razor components and the necessary services.
Three distinct approaches were initially presented. The first was the Single Page Approach, where a single Razor component, for example, StoryCreateEdit.razor, would dynamically adapt its behavior based on a route parameter like StoryId. If StoryId was present, it would operate in "Edit" mode, fetching existing story data; otherwise, it would be in "Create" mode, initializing a new story model. This approach would rely on a single IStoryService with methods like GetStoryForEditAsync, CreateStoryAsync, and UpdateStoryAsync. The second option was the Shared Form Component Approach, which advocated for better separation of concerns. Here, a dedicated StoryForm.razor component would handle only the display of form fields, while two separate parent pages, CreateStory.razor and EditStory.razor, would use this shared form. These parent pages would handle the mode-specific logic and call the same IStoryService. The third approach was the Wizard Approach, suggested for more complex creation processes. This would involve a multi-step wizard orchestrated by a StoryWizard.razor component, with each step having its own child component. A scoped IStoryWizardStateService would maintain data across steps, and the IStoryService would be called at the end. At this stage, no specific approach was rejected, as the goal was to present options.
The discussion then moved to whether a Razor component could have multiple page directives. The answer was yes, confirming that a single component could handle multiple routes like "/stories/create" and "/stories/edit/{StoryId:int}". This directly supported the feasibility of the initial "Single Page Approach".
A critical architectural question was then posed: could rich domain models, specifically EF Core model classes like Story.cs, be used directly as view models in the Blazor UI, given that Data Transfer Objects (DTOs) would be used at the service boundary? The strong recommendation was to avoid this practice. The reasons cited were the risk of leaking persistence concerns into the UI, potential security risks from over-posting (where a malicious user could try to modify properties not intended for client-side editing), the need for UI-specific state that doesn't belong in a domain model, and the increased complexity and payload size. Instead, the best practice was to create dedicated view model classes (e.g., StoryEditViewModel) within the UI project, with client-side mapping between DTOs and these view models. To support this, the IStoryService interface was suggested to be updated to use StoryEditDTO and StoryCreateDTO for data transfer.
Following this, the conversation explored whether the IStoryService should be separated into IStoryReadService and IStoryWriteService. This was strongly recommended, adopting the Command Query Responsibility Segregation (CQRS) pattern. The benefits highlighted were clearer intent, more focused interfaces, better scalability (potentially allowing different data stores or optimizations for read vs. write operations), and improved security. The IStoryWriteService would include an UpdateStoryAsync method. The role of rich domain models outside of EF Core was clarified: they are primarily used on the server to encapsulate business logic and enforce invariants (business rules that must always be true for an object to be in a valid state). Validation was described as occurring at three layers: client-side (view model for user experience), server-side (DTO for API security), and server-side (domain model for business rule integrity).
A subsequent question challenged the placement of rich domain models in the Core project if they were only used on the server, suggesting they might belong in the web server project instead. The explanation emphasized that the Core project serves as the central, shared contract for the entire application, including client, server, background workers, and test projects. Placing domain models in Core prevents circular dependencies and allows other parts of the system (like worker services or test projects) to access the core business logic without depending on the web server project. The cost of including rich domain models in the client's WebAssembly (WASM) payload was also discussed. It was clarified that if the client code uses these models, the .NET ILLinker (tree shaking) would include them and all their transitive dependencies, leading to a significantly larger download size and slower load times. If not used, they are trimmed. This reinforced the need for dedicated view models on the client. Client-side Program.cs was updated to register the new IStoryReadService and IStoryWriteService.
The discussion then revisited the idea of instantiating rich domain models in Razor components for client-side validation reuse. While acknowledging the appeal of a single source of truth for validation, this approach was firmly deemed an anti-pattern. The cons were extensive: massive dependency bloat in the client payload, leaking server-side concerns (like database dependencies or security context) into the UI, a mismatch between UI-specific state and business state, and architectural coupling that makes the system brittle. The conclusion remained that layered validation (ViewModel for UX, DTO for API gateway, Domain Model for integrity) was the superior approach.
A compromise was then proposed: using a shared interface (e.g., IStoryProperties) implemented by both the ViewModel and the EF Core model, with validation rules implemented in static extension methods in the Core library. This was recognized as an "excellent, clever, and viable compromise." Its pros included excellent code reuse, keeping the client payload lean (as only the interface and static methods are referenced, not the full domain model), and decoupling validation from specific models. However, it was noted that this approach wouldn't solve all validation (e.g., cross-model or set-based rules still require server-side context), could lead to a more anemic design (logic outside the objects), and might require custom integration with Blazor's EditForm. Despite these considerations, it was seen as a strong solution for complex, self-contained validation rules. The IStoryOverviewService and HttpStoryOverviewService were marked for removal as they became obsolete with the new CQRS-style services.
This led to a clear articulation of a three-tiered validation strategy:
1.
Property-level validation using DataAnnotations on ViewModels for immediate user experience feedback.
2.
Intra-object invariants involving multiple properties, implemented using a shared interface (like IStoryProperties) and static extension methods, working across both ViewModel and EF Core model.
3.
Cross-model class validations, requiring full database context, implemented exclusively on the server. This analysis was endorsed as "excellent" and "pragmatic." Client-side HttpStoryReadService.cs and HttpStoryWriteService.cs were created to implement the new service interfaces.
The role of DTOs within this three-tiered validation was clarified. DTOs serve as the "language" between the client and server, acting as simple data containers. For reading, the server maps complex domain models to lean DTOs, which the client then maps to ViewModels. For writing, the client maps ViewModels to DTOs, which the server then uses to construct and validate domain models. This prevents over-posting and maintains stable API contracts.
The question of whether ViewModels, DTOs, and EF Core models need to be distinct classes was revisited. The answer remained a firm yes, citing benefits in serving specific masters (UI, network, database), decoupling, and performance. The idea of DTOs using inheritance (e.g., StoryEditDTO inheriting from StoryCreateDTO) was advised against, as it can make DTO contracts less explicit and more brittle. Server-side implementations for IStoryReadService and IStoryWriteService (DbStoryReadService.cs and DbStoryWriteService.cs) were then provided, using IDbContextFactory for proper DbContext lifetime management.
The best practice for accessing the DbContext in server-side services was confirmed to be using IDbContextFactory. This ensures thread safety and controlled, short-lived DbContext instances for each operation. The DbStoryReadService.cs and DbStoryWriteService.cs were updated to consistently use IDbContextFactory.
The discussion then clarified that read DTOs do not need to implement the common write-path interface (IStoryProperties). Read DTOs are created directly from the database via projection, are highly specialized for specific views, and do not require the same validation or mapping logic as write-path objects. Other uses for the shared write-path interface were identified, including reusable UI components, generic helper functions (e.g., logging), and simplified unit testing. A small refinement was made to DbStoryReadService.cs to use null-coalescing operators for robustness in DTO projections.
The philosophical question of whether composition deserves a place alongside encapsulation, inheritance, and polymorphism as a core OOP principle was addressed. The answer was a definitive yes, emphasizing that "Composition over Inheritance" is a fundamental evolution in modern OOP design, promoting flexibility, avoiding rigid hierarchies, and leading to clearer designs.
The conversation then shifted to project organization, specifically whether to group files in the Core project by technical type (e.g., all Models in one folder) or by feature (e.g., all Story-related files in a "Story" folder). While acknowledging the user's current database-first approach, grouping by feature (vertical slicing) was deemed the superior practice for maintainability and scalability in modern applications. The challenge of EF Core tooling finding models in feature folders was addressed by recommending the use of IEntityTypeConfiguration<T> classes, which EF Core can discover automatically via ApplyConfigurationsFromAssembly.
It was confirmed that EF Core tooling does not technically require model classes to be in one folder; discovery is based on DbSet<T> properties in the DbContext. This opened the door for an incremental refactoring strategy: slowly moving model classes and their associated DTOs, interfaces, and mappers into feature-specific folders within Core.
The topic of slugs was introduced, specifically whether they should be user-editable. The best practice was identified as "Generate and Lock": automatically generate the slug from the title, allow initial editing before the first save, but then lock it down once the story is published. This prioritizes URL stability and SEO over continuous user customization, as changing a slug after publication leads to link rot and SEO penalties. The IEditableStoryProperties interface, DTOs, and DbStoryWriteService.cs were updated to reflect this pattern.
A follow-up question challenged the importance of slug stability if the Blazor router uses the StoryId. It was clarified that while the Blazor router might use only the ID, search engines, social media, and user bookmarks treat the entire URL string as the permanent address. Changing the slug breaks this "contract with the internet," leading to severe SEO damage and broken links. The "Generate and Lock" pattern was re-emphasized.
The user then proposed simply not allowing the user to edit the slug at all. This was deemed a perfectly valid and robust architectural choice, prioritizing simplicity and absolute URL stability. The implementation would involve the slug being purely server-generated during creation and never updated by the client. The IEditableStoryProperties interface, DTOs, and DbStoryWriteService.cs were simplified by removing any client-side slug editing capabilities.
The naming convention for the common write-path interfaces was discussed. The recommendation was to use IEditable<Entity>Properties (e.g., IEditableStoryProperties) to explicitly communicate the interface's intent for editable properties. The interface and its implementing DTOs were updated accordingly.
A crucial point of contention arose regarding the properties in IEditableStoryProperties: should they be get-only or have setters? The user noted that the ViewModel would have setters, but the EF Core model's properties (like Title, which is on StoryListing) are not simple get/set fields. It was confirmed that the interface must have setters to fulfill its purpose as an "editable" contract for shared logic. The solution for the Story model was to use explicit interface implementation for these properties, where the getter and setter would correctly access and modify the underlying navigation properties (e.g., StoryListing.StoryTitle). This allows the Story model to conform to the interface without exposing confusing public properties. It was also clarified that the Story model itself should not implement the IEditableStoryProperties interface for collections like StoryTags directly, as the explicit implementation handles this.
The conversation then focused on the StoryTags list conversions in StoryMappers.cs. It was identified that the previous implementation had subtle inconsistencies. The StoryTagDTO class was updated to correctly implement the IStoryTag interface. The mappers (ToStoryUpdateDTO and ToCreateStoryDTO) were simplified to directly use List<IStoryTag> and create defensive copies, ensuring type safety and correct behavior.
The use of exceptions for validation was discussed. While technically possible for client-side validation, throwing exceptions for routine validation failures was deemed an anti-pattern due to performance overhead and fighting Blazor's built-in validation system. The world-class solution was to use a List<string> of errors for client-side UX feedback, and for the server-side service to throw a custom StoryValidationException (containing the list of errors) when validation fails. This provides shared validation logic but applies it appropriately for each environment. The IStoryWriteService and DbStoryWriteService.cs were updated to reflect this exception-based server-side validation.
Finally, the creation of the ViewModel class and a reusable Razor component for the form was addressed. A StoryPropertiesViewModel.cs was created in SharedUI, implementing IEditableStoryProperties and including DataAnnotations for client-side validation. A corresponding StoryPropertiesForm.razor component was created, designed as a reusable "dumb" component that binds to this ViewModel. The form included basic input fields and validation messages. The TagSelector.razor component was then introduced as a self-contained, reusable component for tag selection, encapsulating all tag retrieval and selection logic. The StoryPropertiesForm.razor was updated to host multiple instances of this TagSelector component for different tag types.
The explanation of tags in StoryPropertiesForm.razor detailed the purpose of Razor directives like @using, Blazor components like EditForm, DataAnnotationsValidator, InputText, InputTextArea, InputSelect, and ValidationMessage, and the @code block parameters. It was clarified that Microsoft.AspNetCore.Components.Forms is the correct namespace for these components, and it's typically imported globally via _Imports.razor.
The use of blanket @using statements in the root _Imports.razor for SharedUI and Core libraries was confirmed as standard and recommended best practice for Blazor applications, as it significantly improves productivity and component cleanliness. A minor code quality suggestion was made to DbStoryReadService.cs to use direct projections for optimal query generation in GetStoryForEditAsync.
Finally, the issue of StoryPage.razor having trouble recognizing classes despite global imports was addressed. The explanation highlighted a common C# rule: a using statement for a parent namespace does not automatically include its sub-namespaces. The solution was to explicitly add @using directives for specific sub-namespaces like TheCanalaveLibrary.Core.Story and TheCanalaveLibrary.Core.ServiceInterfaces to _Imports.razor.

Your instinct is spot-on. Forcing a complex autocomplete component on a list of only 10 items is poor UX, just as a dropdown with 2000 items is unusable.
The Architectural Decision: One Component or Two?
The best practice in this scenario is to create one smart, self-contained component that can adapt its presentation. This approach is superior because:
1.
Simplicity for the Consumer: The parent component (StoryPropertiesForm) doesn't need to know about the different presentation modes. It just needs to say, "I need a tag selector for 'Genres'," and the component itself handles the rest. This keeps the parent component clean and simple.
2.
Encapsulation: All the logic for fetching tags, managing state, and deciding how to render is encapsulated within the TagSelector component. This makes it a true "black box" and easy to reuse anywhere.
3.
Maintainability: If you decide to change the threshold for switching between a dropdown and an autocomplete, you only have to change it in one place: the TagSelector component.
The World-Class Implementation
Based on your excellent proposal, I have created a new, more powerful TagSelector.razor component. This single component will:
•
Accept a TagType to know which tags to fetch.
•
Accept a DisplayThreshold parameter. If the number of available tags is above this threshold, it will render the autocomplete input. Otherwise, it will render a more user-friendly list of checkboxes.
•
Bind to the parent's list of selected tags, encapsulating all the complex interaction logic.
