A Chronicle of Building a Hybrid Blazor Application
Our journey began with a simple request to understand how to open a project file in the Rider IDE. The initial advice was straightforward, focusing on opening the main solution file, which in this case was TheCanalaveLibrary.sln, rather than individual project files. This ensures that the entire context of the application, with all its interconnected projects, is loaded correctly. From there, the discussion moved to viewing the raw XML of a project file. The standard method was to right-click on a project, such as TheCanalaveLibrary.SharedUI, and select the edit option, which opens the underlying csproj file directly in the editor.
A key architectural decision point arose when you wanted to add a framework reference to Microsoft.AspNetCore.App. You asked if there was a user interface tool for this, but it was explained that modifying framework references is a fundamental change that, by convention in the .NET ecosystem, is done by directly editing the project's XML file. This established a pattern of making precise, deliberate changes to the project configuration files.
The conversation then turned to the core challenge of building a modern, hybrid Blazor application using the cutting-edge .NET 10 framework. The initial goal was to have pages from a shared class library, TheCanalaveLibrary.SharedUI, render correctly within the main server application, which was at that time named TheCanalaveLibrary and later renamed to TheCanalaveLibrary.Server. This led to a persistent 404 Not Found error for pages defined in the shared library, even though pages defined directly in the server project, like the login page, worked perfectly.
The troubleshooting process was a deep and iterative exploration of Blazor's architecture. The first major breakthrough came from understanding that the Blazor Router, defined in the Routes.razor component, is responsible for discovering all routable pages. While it automatically scans its own project's assembly, it must be explicitly told to scan other assemblies. The solution was to add the AdditionalAssemblies parameter to the Router component, providing it with the assemblies of the TheCanalaveLibrary.Client and TheCanalaveLibrary.SharedUI projects. To make this reference clean and stable, we created empty marker components, WasmClientAssemblyIdentifier.razor and SharedUIAssemblyIdentifier.razor, in their respective projects to serve as reliable type references for identifying the assemblies.
However, the 404 error persisted. This led to the discovery of a second, equally important principle of modern Blazor routing. The main ASP.NET Core endpoint mapping system, configured in Program.cs via the app.MapRazorComponents method, also needs to be aware of the external assemblies. The server logs, which you enabled by modifying appsettings.Development.json to increase the log verbosity for Microsoft.AspNetCore.Routing and Microsoft.AspNetCore.Components, provided the critical clue. The message "Request did not match any endpoints" proved that the server itself, not just the Blazor router, was unaware of the route for the home page. The fix was to also call the AddAdditionalAssemblies method on the app.MapRazorComponents call, ensuring that the server's master route table was correctly populated at startup.
With static rendering from the shared library finally working, the focus shifted to enabling interactivity. The goal was to create a high-performance application that was "static by default" but allowed individual pages to opt-in to either server-side interactivity via SignalR or client-side interactivity via WebAssembly. A key point of confusion arose around the methods AddInteractiveServerRenderMode and AddInteractiveWebAssemblyRenderMode. The official Microsoft documentation states that the first of these calls sets a global default render mode for the entire application. Your own experiment, however, provided the crucial evidence that this default does not automatically apply to routable pages discovered in external libraries; they remain static unless they explicitly opt-in with an at-rendermode directive. This confirmed that the default template's configuration in Program.cs was indeed the correct and canonical way to build a flexible, opt-in hybrid application. It enables the server capabilities for interactivity by registering the necessary endpoints without forcing interactivity on every page.
The syntax for the rendermode directive itself was another learning point. Initial suggestions incorrectly used a simple token like InteractiveServer. The correct syntax, as you discovered through compiler errors, requires providing a static object instance from the RenderMode class, such as RenderMode.InteractiveServer or RenderMode.InteractiveWebAssembly.
Switching to WebAssembly interactivity introduced a new build error, NETSDK1082, which stated that the server-only Microsoft.AspNetCore.App framework was not available for the browser-wasm runtime. This happened because, to solve an earlier server-side compilation error, we had added a FrameworkReference to Microsoft.AspNetCore.App in the TheCanalaveLibrary.SharedUI.csproj file. This made the shared library incompatible with the client project. The solution was to remove the FrameworkReference and replace it with a PackageReference to the specific, platform-agnostic Microsoft.AspNetCore.Components.Web package. This provided the SharedUI library with the essential Blazor APIs it needed, like the RenderMode class, without tying it to the entire server framework, thus making it truly isomorphic and shareable.
Finally, with all core functionality working, we addressed the last remaining visual bug: a persistent "unhandled exception" message from the blazor-error-ui div in the MainLayout.razor file, even on static pages with no errors. The server logs were clean, and browser console errors related to a missing favicon.png and incorrect script paths were resolved by placing the icon in the wwwroot folder and using standard HTML link and script tags in the application's root component. The final diagnosis for the error banner was a client-side JavaScript initialization issue. On a purely static page, the HTML for the ReconnectModal component is not rendered. However, its associated JavaScript file, ReconnectModal.razor.js, was still being loaded and would throw an error when it tried to find the modal's HTML element in the document, which did not exist. The Blazor runtime would catch this script error and display the fatal error banner. The robust, production-ready solution was to conditionally render the ReconnectModal component and its script only on pages that have an interactive render mode, using a check in the App.razor component. This required making the HttpContext available via a CascadingParameter to check the endpoint's metadata. This final change ensured that interactivity-related UI and scripts were only loaded when absolutely necessary, preserving the true "static by default" architecture and resolving the last outstanding issue.
