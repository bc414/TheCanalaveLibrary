Building a Responsive and Maintainable Blazor Application
Our conversation began with a clear and practical goal: to create a Blazor application that could serve different layouts and content depending on whether the user was on a desktop or a mobile device. The initial approach we discussed was based on using JavaScript to detect the device type. This involved creating a JavaScript file named device.js, which contained a function called isMobile. This function would use the browser's window.matchMedia API to check if the screen width was below a certain threshold, such as 768 pixels, returning true if it was. To make this accessible within the Blazor application, we created a C# class named DeviceDetectionService. This service injected Blazor's IJSRuntime, which allows C# code to call JavaScript functions. The service had a single method, IsMobile, that would call the JavaScript isMobile function. We then registered this service with the application's dependency injection container. The final piece of this initial plan was to create a layout component, DeviceLayout.razor, that would inject our new service, call its IsMobile method, and then use a simple if-else block to render either a DesktopLayout.razor or a MobileLayout.razor component, passing along the page's content.
The discussion then evolved to refine this structure. You asked how to properly use an interface for the service, a key principle for building loosely coupled applications. We defined an interface named IDeviceDetectionService and had the DeviceDetectionService class implement it. The dependency injection registration in the Program.cs file was updated to map the interface to the concrete class. This meant that components would now depend on the IDeviceDetectionService interface, not the specific implementation.
A critical insight then changed our direction. You correctly identified that calling IJSRuntime would fail during the initial static server-side rendering of a Blazor Web App, as there is no active browser connection at that stage. This led to our first abandoned approach. The initial fix we discussed was to modify our components to handle this prerendering lifecycle. Instead of calling the service in the OnInitializedAsync lifecycle method, we would move the call to OnAfterRenderAsync. This method only runs after the component has been rendered in the browser and an interactive connection is established. The drawback of this approach, however, was that the page would initially render without knowing the device type, requiring a loading message or placeholder, which could cause a flicker or layout shift once the device type was determined.
Recognizing this user experience issue, you asked how a purely statically rendered page could perform this detection without JavaScript. This led to a much more robust solution. We modified the DeviceDetectionService to be "smarter." It would now inject not only IJSRuntime but also IHttpContextAccessor. The logic was updated so that if it was running on the server (where HttpContext is available), it would inspect the User-Agent string from the incoming HTTP request headers to check for a mobile identifier like "Mobi". If it was running on the client in a WebAssembly context (where HttpContext is null), it would fall back to using the original IJSRuntime method. This was a significant improvement because it allowed the detection to happen on the server during the initial static render, eliminating the flicker and the need for the OnAfterRenderAsync logic.
Building on this, you proposed an even cleaner architectural pattern. Instead of a single service with conditional logic, why not have two distinct implementations? This was an excellent idea that we adopted as the final pattern for device detection. We created two classes that both implemented the IDeviceDetectionService interface. The first was ServerDeviceDetectionService, which lived in the main server project and used IHttpContextAccessor to check the User-Agent. The second was WasmDeviceDetectionService, which lived in the client-side project and used IJSInProcessRuntime to perform a synchronous JavaScript call. Each project's Program.cs file was configured to register its own specific implementation for the shared IDeviceDetectionService interface. This cleanly separated the server and client concerns.
During this process, we encountered and solved a few specific problems. When testing the WebAssembly implementation, you received a JavaScript error that the isMobile function was not defined. This was because the device.js file, located in the SharedUI Razor Class Library, was not being loaded by the main application. We solved this by adding a script tag to the main App.razor file, using the special Blazor path syntax underscore-content-slash-TheCanalaveLibrary.SharedUI-slash-js-slash-device.js to correctly reference the static asset from the shared library. We also discussed how to test the functionality using browser developer tools, which allow you to simulate a mobile device by changing both the screen size and the User-Agent header, and how to verify the Wasm implementation specifically by disabling prerendering on a page.
The conversation then shifted from device-specific layouts to device-specific pages. You realized that as the application grew, putting if-else logic in every page to show different content would be unmanageable. This led us to establish the "Dispatcher Page" pattern. We organized the project by creating Desktop and Mobile subfolders inside the Pages directory. For a given route, like the home page, the component with the at-page directive, Home.razor, would become a simple dispatcher. Its only job would be to inject the IDeviceDetectionService and then render either a HomeDesktop.razor or a HomeMobile.razor component. These platform-specific components would contain the actual UI and logic, keeping them clean and focused.
We then applied this architecture to a real-world data service. You asked about naming conventions, and we confirmed that your proposed structure of an interface like IStoryOverviewService, a server implementation named StoryOverviewService, and a client implementation named HttpStoryOverviewService was an industry best practice. We encountered another technical hurdle here: an "ambiguous invocation" error on a LINQ query. After first checking for a missing using System.Linq statement, we diagnosed the root cause as a dependency conflict. Your TheCanalaveLibrary.Core project, which was shared between the server and client, had a reference to the full, server-only Microsoft.AspNetCore.Identity package. This created ambiguity for the WebAssembly compiler. We resolved this by refactoring the Core project to reference the much lighter Microsoft.Extensions.Identity.Stores package, which provided the necessary classes without the heavy server-side dependencies.
With the architecture established, we discussed event handling. We defined the "Smart Component and Dumb Component" pattern, where the dispatcher page is the "smart" component that handles data fetching and shared state logic, while the presentation components are "dumb," simply receiving data and raising events via EventCallback. You also asked how to add interactivity, like a button click, to an otherwise statically rendered page. This introduced the concept of "islands of interactivity." The best practice, we determined, was to create a very small, dedicated component for the interactive element, such as a DeleteStoryButton.razor, and apply an interactive render mode like at-rendermode RenderMode.InteractiveServer just to that component. The static parent page would then use this interactive component, which would communicate back up using an EventCallback.
To test the full client-to-server communication path, we implemented a GetRandomNumber method on the IStoryOverviewService. This involved creating a minimal API endpoint on the server. You correctly pointed out that to truly test the client path, the component needed to use InteractiveWebAssembly mode, not InteractiveServer. We made this change, which forced the client-side HttpStoryOverviewService to be used, which in turn called the server's API endpoint. You then asked how to keep the Program.cs file from getting cluttered with API endpoints. We refactored this by creating a StoryEndpoints.cs file containing an extension method, MapStoryEndpoints, which encapsulated all the routing logic for that feature. The Program.cs file then just needed a single line to call this method.
Our conversation concluded with a deep dive into the nature of C# extension methods. You astutely observed that they seem to "flip" the caller and the first parameter. I confirmed this was an excellent analogy for the syntactic sugar that the compiler applies, translating a call like app.MapStoryEndpoints() into the static method call StoryEndpoints.MapStoryEndpoints(app). We explored how this compile-time feature differs from the runtime "monkey-patching" in dynamic languages like JavaScript. Finally, you drew a powerful conclusion: that extension methods, rather than violating object-oriented principles, are actually a more "raw" and explicit representation of how instance method calls work under the hood. I confirmed this insight, explaining that the compiler's handling of an instance method call, where it passes the object's this reference as a hidden first argument (typically via a CPU register), is conceptually identical to how an extension method is called. This design allows extension methods to provide fluent syntax while fully respecting OOP principles like encapsulation, as they can only access the public members of the type they are extending.
