A Detailed Chronological Record of an EF Core and PostgreSQL Implementation
Our conversation began with a common but tricky issue in a .NET application using Entity Framework Core: a persistent warning message indicating that the database model was changing on every build. This warning, specifically the Microsoft.EntityFrameworkCore.Migrations.PendingModelChangesWarning, is a safety mechanism to prevent applying migrations to a database when the application's current model doesn't match the migration's snapshot.
The initial investigation focused on the ApplicationDbContext.cs file. My first suggestion was that the problem might be caused by the use of the HasDefaultValueSql method with the string "CURRENT_TIMESTAMP" on properties within a Table-Per-Type inheritance hierarchy. This was observed on several derived comment models, such as BlogPostComment and ChapterComment, which inherited from a BaseComment class. The proposed solution was to consolidate this configuration by moving the default value setting to the DatePosted property on the base BaseComment entity's configuration. The reasoning was that this would define the default value only once, creating a more stable model.
This initial advice was quickly corrected by you. You clarified that you had intentionally designed the database schema to denormalize the DatePosted column, placing it on each specific comment table to improve query performance by avoiding joins. My first suggestion would have undone this optimization. Acknowledging this, I proposed a second approach: to keep the DatePosted property in the base C# model, BaseComment.cs, but to configure Entity Framework Core to ignore it on the base table and map it only to the derived tables. My first attempt at this involved suggesting the use of the Ignore method in the Fluent API, which was an incorrect application of that feature. A subsequent refinement suggested using the IsMapped method, setting it to false for the base entity and true for the derived entities. This also proved incorrect, as you rightly pointed out that it caused a compilation error.
This led to the fourth and ultimately correct solution for this part of the problem. The proper way to achieve this denormalization in Entity Framework Core was to define the DatePosted property on the base C# model, BaseComment, and then, in the ApplicationDbContext's OnModelCreating method, to provide an explicit configuration for that property on each of the derived entity configurations, like BlogPostComment. By configuring the property on the derived types, we override Entity Framework Core's default convention of mapping it to the base table. This correctly instructs the framework to create the column on each specific child table, preserving your performance-oriented schema while also stabilizing the model.
With that issue seemingly resolved, the conversation shifted to a different part of the ApplicationDbContext.cs file. You asked about a global configuration loop that set all DateTime properties to use a specific precision, a practice you had carried over from a previous implementation using SQL Server. You questioned if this was still necessary now that you were using PostgreSQL. I explained that unlike SQL Server, PostgreSQL's timestamp data type uses a fixed 8 bytes of storage regardless of precision, so the configuration was no longer providing its original space-saving benefit. I also noted that a similar loop for the DateOnly type was redundant because the Npgsql Entity Framework Core provider handles that mapping correctly by default. I recommended removing both loops to simplify the code.
Next, you asked how to verify which connection string the dotnet ef database update command was using in the terminal. I outlined several methods, with the most direct being the use of the verbose flag. Running the command with this flag would print detailed logs, including the exact server and database being targeted. I also described how to explicitly pass a connection string using the connection argument and how to check for relevant environment variables like ASPNETCORE_ENVIRONMENT, which determines which appsettings file is used.
Despite these changes, you reported that the original PendingModelChangesWarning persisted. This indicated another, more subtle source of instability. My next suggestion was to refactor the OnModelCreating method. I observed many separate configuration blocks for setting default timestamp values and hypothesized that the non-deterministic order in which Entity Framework Core might process these fragmented configurations could be the cause. I recommended consolidating all the HasDefaultValueSql calls into one contiguous block of code.
When even this did not resolve the issue, a deeper review of ApplicationDbContext.cs revealed the true root cause. The problem was in the HasData calls used for seeding lookup tables. You were seeding data using new instances of the entity classes themselves, for example, new NotificationType. I explained that this is a known source of model instability. The recommended best practice is to use anonymous types for all HasData calls, for instance, new followed by curly braces containing the property assignments. This provides a more deterministic structure for Entity Framework Core's model builder.
This change successfully resolved the long-standing instability warning, but it introduced a series of new, more straightforward compilation and migration errors that we worked through systematically. The first was a type mismatch error stating that an integer value could not be used for a property of type short, specifically for the AcknowledgmentRoleId in the AcknowledgmentRole entity. I explained that the C# compiler infers a numeric literal like 1 as an integer, and the fix was to explicitly cast these values to short within the anonymous type definition in the HasData call. You then proactively asked if other such casts were needed, and upon review, I identified the same issue in the seed data for RecommendationStatus, ReportReason, StoryRelationshipType, and Theme, providing the necessary casts for all of them.
The next error was that no value was provided for the required property DefaultCollapsed in the NotificationType seed data. Because this was a non-nullable boolean property, the fix was to add the DefaultCollapsed property assignment, with a value of false, to every entry in the NotificationType seed data.
With the model finally building correctly, the database update command began to execute but failed with a PostgreSQL unique violation error. The logs showed it was failing to create a unique index on the display_name column of the notification_types table. The cause was duplicate values in the seed data; for example, two different notification types were both named "New Follower". The solution was to make these display names more specific and unique, such as "New Story Follower" and "New Profile Follower".
After successfully applying the initial migration, you began inspecting the generated database schema and had several architectural questions. You noticed that the ASP.NET Identity tables were still named with PascalCase, while your custom tables were correctly using snake_case. You asked if this would cause problems. I explained that it would not, as the base IdentityDbContext from which your ApplicationDbContext inherits explicitly configures these table names, and the UseSnakeCaseNamingConvention method is designed to not override such explicit configurations. I assured you that Entity Framework Core would manage these different naming schemes transparently.
You also observed two "pure junction tables" that were created by convention for many-to-many relationships, specifically story_character_story_character_relationship and group_folder_group_story. You asked how these would be used. I explained that you would never interact with these tables directly in your C# code. Instead, you would manipulate the ICollection navigation properties on your C# models, and Entity Framework Core would automatically translate those actions into the necessary SQL insert and delete statements on these hidden junction tables.
The conversation then moved to advanced architectural patterns. You described a plan to have a background worker refresh the data in the also_favorited_scores table and wanted to ensure zero downtime. I advised against a simple truncate-and-insert strategy and instead recommended the "Staging and Swap" pattern. This involves creating a staging table, populating it in isolation, and then using an atomic RENAME operation within a database transaction to instantaneously swap the live table with the new one. You asked a very insightful follow-up question about what would happen if an application thread tried to update a row at the exact moment of the swap. I explained the database locking scenarios that would occur, which would likely result in either a concurrency exception or a "wasted write" to the old table. This confirmed that your plan to make the table read-only from the application's perspective was the correct and safest design.
This led to a discussion about performance. You asked if there was a way to make the DbContext read-only to improve performance and memory usage. I explained the overhead of Entity Framework Core's change tracker and introduced the AsNoTracking method for per-query optimizations. For a more robust solution, I recommended the "Read-Only DbContext Pattern". This involved creating a new ReadOnlyApplicationDbContext class that inherits from your main ApplicationDbContext but sets the default query tracking behavior to no-tracking. I showed how to register this in the Program.cs file and outlined the architectural rule: inject the read-only context for services that only read data, and inject the standard, writable context for services that perform writes. For services that do both, the best practice is to inject only the standard context and use the AsNoTracking method for its read-only queries. I also clarified that when using this two-context pattern, migration commands must explicitly specify the writable context using the context argument.
Finally, you asked whether you should check your migration files into source control, given that you were frequently deleting and recreating them during prototyping. I strongly recommended that you do. I explained that even in the prototyping phase, the migration file serves as the single source of truth for your schema, acting as both documentation and a crucial backup. Committing it establishes a professional workflow that is essential for collaboration and for creating repeatable, ordered database upgrades once the application moves toward production.
